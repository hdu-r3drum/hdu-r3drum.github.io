大文件查重思路

Edwards Lv2

问题

给定A，B两个文件，各存放50亿个url，每个url各占64字节，内存限制是4G，如何找出这两个文件中的共同url？

分析问题

数据量

每个 URL 64 字节，即 50 亿 * 64B = 320GB，单个文件大小约 320GB，两个文件共640GB。
远超4GB内存，无法直接加载进内存。

目标

找到 A 和 B 之间的公共 URL。

方法 1：基于哈希分桶（External Hashing）

由于 64 字节的 URL 可以直接哈希，可以采用哈希分桶+归并的方法。

步骤

  • 遍历文件 A，对 URL 进行哈希分桶
    使用哈希函数 hash(url) % N，将 A 的 URL 分成 N 个小文件（A_0, A_1, ..., A_(N-1)）。
    例如，N = 1000，每个小文件 A_i 约 320MB（320GB / 1000）。
  • 遍历文件 B，按照相同规则进行哈希分桶
    处理 B，生成 B_0, B_1, ..., B_(N-1)。
  • 逐个比对 A_i 和 B_i
    加载A_i到内存中的哈希集合（HashSet），占用最多 320MB。

逐行读取 B_i，检查是否在 A_i 的哈希集合中。
输出交集 URL 到结果文件

方法 2：基于外部排序（External Merge Sort）

如果无法哈希（如 URL 无法均匀分布），可以使用外部排序+归并。

步骤

  • 对A和B进行外部排序
    分块排序：每次读取100MB数据到内存，使用快排或归并排序，然后写入有序临时文件。
    例如，A 和 B 各产生 3200 个有序临时文件（假设 100MB/块）
  • 多路归并
    使用最小堆（优先队列）进行多路归并，合并所有 3200 个文件，得到排序后的 A' 和 B'。
  • 双指针扫描 A’ 和 B’ 找交集
    依次读取A'和 B'：
    若 A[i] == B[j]，输出到结果文件。
    若 A[i] < B[j]，i++，否则 j++。

方法 3：Bloom Filter（适用于 B 远小于 A）

如果B较小（例如 B 只有 1亿 URLs），可以使用 布隆过滤器（Bloom Filter） 提高效率。

步骤

  • 构建 Bloom Filter
    使用 布隆过滤器（BF） 存储 B 的 1亿 URLs。
    64GB URL，误判率 0.01% 需要约 1GB 内存，可以接受。
  • 遍历 A，检查 B
    遍历 A，对每个 url 查询 Bloom Filter 是否存在于 B 中。
    命中则存入结果文件。

总结

方法 适用场景 时间复杂度 空间需求 额外磁盘操 适用数据特点
哈希分桶 A 和 B 相似大小 O(N) O(N/Bucket) 少 哈希均匀分布
外部排序 任意情况 O(N log N) O(块大小) 多 适用于无规律 URL
布隆过滤器 B 远小于 A O(N) O(1GB) 无 误判可接受

若 A、B 大小相近 → 哈希分桶
若 URL 无法哈希均匀 → 外部排序
若 B 远小于 A → Bloom Filter
最佳方案：哈希分桶法（外部哈希 + 归并），因为它 时间复杂度低，IO 读写少，适用于一般场景！
